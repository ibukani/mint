# ADR 001: 静的 Feature-Module 設計の採用

## ステータス
Accepted

## コンテキスト
Mint は複数の独立した便利ツール（時計、音声入力など）を束ねるデスクトップアプリケーションです。
将来的に多数のツールが追加されることが想定されます。
これらのツールをどのようにシステムに統合するかが課題となりました。

選択肢として以下の2つがありました：
1. **動的プラグインアーキテクチャ**：実行時にDLLやWASMモジュールをロードし、汎用JSONで通信する。
2. **静的 Feature-Module アーキテクチャ**：コンパイル時に全てのモジュールを静的にリンクし、専用の型で通信する。

## 決定
**静的 Feature-Module アーキテクチャ** を採用します。

## 理由
- **型安全性**: 動的ローディングや汎用JSONでは、TypeScriptとRust間の型の不整合が実行時エラーとして顕在化しやすくなります。静的設計であれば、ビルド時に全てのエラーを検知できます。
- **AIエージェント親和性**: AIが自律的に機能追加を行う際、型の恩恵を受けられる静的言語環境のほうが、AIが文脈を理解しやすく安全です。
- **複雑性の排除**: 個人用ツールプラットフォームにおいて、動的プラグインのような高度な抽象化はオーバーエンジニアリングであり、保守コストに見合いません。

## 結果
- 新規ツールの追加には `npm run scaffold:feature` によるコード生成と、一部の手動配線が必要になります。
- 実行時のプラグイン追加はできず、追加には再ビルドが必要になります（許容できるトレードオフ）。
