# AI 開発基盤ガイドライン (AI Development)

AI エージェントが Mint リポジトリで開発・機能追加を安全かつ自己完結で行うための運用ルールです。

## 1. 作業開始時のステップ
タスクを開始する前に、以下のファイルを必ず確認してください。
- `README.md`: プロジェクトの全体像と機能のステータス（implemented / placeholder）の把握
- `.agents/AGENTS.md`: AI用アーキテクチャルール
- `docs/architecture.md`: システム設計方針
- 既存のテストおよびCIスクリプト (`package.json` 内)

## 2. Feature 追加時の標準フロー
1. 新機能が必要と判断した場合、`npm run scaffold:feature <feature_name>` を実行。
2. 生成されたフロントエンド/バックエンドファイルを要件に合わせて実装。
3. `src/core/mocks/tauriMock.ts` へのモック定義追加。
4. `src-tauri/src/lib.rs` へのコマンド登録。
5. （ウィンドウが必要な場合）`tauri.conf.json` および `src/core/windowRoutes.ts` の編集。

## 3. 検証と再修正ループ
コードを変更した後は、必ず以下のコマンドを実行し、アーキテクチャと型の整合性を確認します。

```bash
npm run check
```
（内部で `tsc`, `biome check`, `vitest`, `verify-architecture.js`, `vite build` が実行されます）

Rustコードを変更した場合、またはフロントエンドからの呼び出し型定義を変更した場合は、以下も実行します。

```bash
npm run check:tauri
```
（Rust非搭載環境で実行不可の場合は、CIに委ねるか、その旨を報告してください）

エラーが発生した場合は：
1. `as any` を使ってエラーを隠蔽しないこと。
2. エラーメッセージから根本原因（型の不一致、モックの登録漏れ等）を特定し、正しく修正すること。

## 4. PR作成前チェックリスト
- [ ] 全ての検証コマンド (`npm run check`) がパスしているか？
- [ ] `docs/` 内のアーキテクチャ資料と `.agents/` 内のルールに乖離が発生していないか？（発生した場合は同期する）
- [ ] Placeholderな機能にアクセスするUIがユーザーを混乱させないか？
- [ ] 新規スクリプトや生成コードがGit管理から漏れていないか？

## 5. AI がやってはいけないこと
- `as any` による型安全性の回避。
- 動的プラグインローダーの実装。
- コンパイルエラーを握り潰す場当たり的な修正。
- ユーザー確認なしの大規模なアーキテクチャ変更。
